# 개발 가이드라인

이 문서는 AI와 협업하여 프로젝트를 개발할 때 따라야 할 원칙과 절차를 정의합니다.

---

## 1. 이슈 처리 프로세스

### 이슈 발생 시 절차

```
0. 프로젝트 구조 파악 (Project_Structure.md)
   ↓
1. 문제 파악
   ↓
2. 여러 해결 방안 모색
   ↓
3. 분석 보고서 작성 (docs/issues/ISSUE-XXX-제목.md)
   ↓
4. 사용자 검토 및 방안 선택
   ↓
5. 선택된 방안으로 구현
   ↓
6. 검증 및 문서 업데이트 (구현된 내용이 반영되어야 하는 모든 문서 대상)
```

### 보고서 필수 항목

- **문제 요약**: 무엇이 문제인지 간결하게 설명
- **원인 분석**: 왜 발생하는지
- **해결 방안 비교표**: 난이도, 영향력, 위험성 평가
- **각 방안 상세**: 장단점, 구현 방법
- **권장 사항**: AI의 추천과 그 이유
- **불확실한 요소**: AI가 판단하기 어렵거나 불확실한 점들

### 보고서 분량

- **5분 이내로 읽을 수 있는 분량**
- 핵심 정보 위주로 작성
- 불필요한 설명 제거

## 2. 변경 관리 원칙

### 즉시 변경하면 안 되는 경우

- 코드 구조에 영향을 주는 변경
- 여러 파일에 걸친 수정
- 기존 아키텍처를 변경하는 작업
- 되돌리기 어려운 변경

### 즉시 변경해도 되는 경우

- 단순 오타 수정
- CSS 스타일 미세 조정
- 주석 추가/수정
- 단일 파일 내 버그 수정

## 3. 코드 작성 원칙

### 파일 구조

```
js/
├── app.js              # 앱 진입점
├── state.js            # 상태 관리
├── pages/              # 페이지 컴포넌트
├── components/         # 재사용 컴포넌트
├── services/           # 데이터/API 서비스
└── utils/              # 유틸리티 함수
```

### 명명 규칙

| 유형 | 규칙 | 예시 |
|------|------|------|
| 파일 (페이지/컴포넌트) | PascalCase | `TeamSetupPage.js` |
| 파일 (유틸/서비스) | camelCase | `constants.js` |
| CSS 파일 | kebab-case | `team-setup.css` |
| 클래스명 | PascalCase | `class ChampionService` |
| 함수명 | camelCase | `function createSlot()` |
| 상수 | UPPER_SNAKE_CASE | `const POSITIONS = []` |

### CSS 규칙

- CSS 변수로 색상/크기 관리
- BEM 명명법 사용: `.block__element--modifier`

## 4. 테스트 절차

### 로컬 테스트 방법

ES6 모듈 사용으로 인해 `file://` 프로토콜에서 직접 열 수 없음.

**권장 방법**:

```bash
# 방법 1: VS Code Live Server 확장 사용
# index.html 우클릭 → "Open with Live Server"

# 방법 2: npx serve
npx serve

# 방법 3: Python
python -m http.server 8080
```

### 테스트 체크리스트

- [ ] 팀 설정 → 챔피언 픽 → 경기 상황 전체 플로우
- [ ] 드래그앤드롭 기능 (배치, 교환, 제거)
- [ ] 검색 기능 (한글/영문)
- [ ] 브라우저 새로고침 후 데이터 유지
- [ ] 콘솔에 에러 없음

## 5. 문서 관리

### 필수 문서

| 문서 | 용도 |
|------|------|
| `PLAN.md` | 개발 계획서 |
| `Project_Structure.md` | 프로젝트 구조와 문서 정보들 |
| `CHANGELOG.md` | 버전별 변경 이력 |
| `DEV_GUIDELINES.md` | 개발 가이드라인 (이 문서) |
| `lessons_learned.md` | 교훈 및 시행착오 기록 |
| `issue.md` | 발생한 이슈 기록 |
| `docs/issues/` | 이슈별 상세 분석 보고서 |

### CHANGELOG 작성 규칙

```markdown
## [버전] - YYYY-MM-DD

### Added
- 새로 추가된 기능

### Changed
- 변경된 기능

### Fixed
- 수정된 버그

### Removed
- 제거된 기능
```

## 6. AI 협업 시 유의사항

### AI에게 명확히 전달해야 할 것

1. 변경의 목적과 배경
2. 제약 조건 (건드리면 안 되는 것)
3. 기대하는 결과

### AI가 따라야 할 것

1. **이슈는 분석 먼저, 해결은 승인 후**
2. 구조적 변경 전에 반드시 확인
3. 변경 사항은 작은 단위로 나누어 진행
4. 각 변경 후 동작 확인
